This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

An iOS port of Anki, in early implementation. **`PLAN.md` is the source of truth** — read it before doing any work. It contains the full architecture (already decided — do not re-evaluate), the target repo layout, five implementation phases with explicit success criteria, and known risks. Work phase by phase; each phase must be green before starting the next.

The Anki desktop source (`ankitects/anki`, branch `main`) is checked out at `/home/ramon/anki` and is the reference for everything. **Do not modify `/home/ramon/anki`** except as allowed in Phase 1 (conditional iOS features in `rslib/Cargo.toml`, on a branch, documented in README).

## Architecture (summary — details in PLAN.md)

Three layers:

1. **SwiftUI shell** (`AnkiIOS/`, generated with xcodegen/tuist — never commit a hand-made `.xcodeproj`): native navigation (deck list, settings, sync) plus `WKWebView` for the existing Anki web pages (reviewer, editor, deck options, stats).
2. **`bridge/` Rust crate** (staticlib): a minimal C ABI (~5 functions passing protobuf bytes: open/close backend, run_method, free_bytes, start mediasrv) over `anki = { path = ../../anki/rslib }`. No UniFFI — the API is protobuf-binary. Swift messages come from `swift-protobuf` over `/home/ramon/anki/proto/anki/*.proto`.
3. **rslib** (unchanged or nearly so): Anki's Rust core, which already embeds the axum web server (mediasrv) that serves the Svelte pages from `ts/`. On iOS it runs in-process on loopback and the WKWebViews load from it.

Web assets are compiled once from the desktop repo (`ts/` → `out/`) into `AnkiIOS/Resources/web/` (gitignored, regenerated by script).

## Rules from the plan

- **Reuse, don't rewrite**: for any feature, first look for the RPC in `proto/anki/` and the page in `ts/` of the desktop repo. Only write Swift for navigation and for logic that on desktop lives in `qt/aqt` (Python).
- Canonical references for integration questions, all in `/home/ramon/anki`: `pylib/rsbridge/lib.rs` (FFI pattern), `pylib/anki/_backend.py` (RPC dispatch and error mapping), `qt/aqt/mediasrv.py` (how desktop consumes the web server), `rslib/src/ankidroid/` (precedent for a non-Python frontend).
- Generated Swift protobuf code is committed; heavy generated artifacts (web assets, `.xcframework`) are not — gitignore plus reproducible scripts in `scripts/`.
- Small commits per phase.

## Environment constraints

- This machine is Linux (WSL2). Phase 1 steps 1–2 can be validated here with `cargo check` (host target, no iOS target). Everything requiring iOS targets, Xcode, or the simulator needs macOS.
- iOS Rust targets: `aarch64-apple-ios` (device) and `aarch64-apple-ios-sim` (simulator), packaged into an `.xcframework`.
